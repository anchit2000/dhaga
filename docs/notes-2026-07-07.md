## The one real code gap I found (§3): embeddings cleanup for note deletion

- `forgetContact` (`apps/web/src/lib/repo/contacts/mutations.ts:68-95`) does a
  full, correct cascade including `deleteEmbeddingsByContact` — no gap there.
- But `deleteNote` (`apps/web/src/lib/repo/notes.ts:43-50`) only tombstones
  notes/facts/edges — it does **not** delete the note's embeddings itself.
  Embeddings cleanup (`deleteEmbeddingsForNote`) only happens one layer up,
  in the server action `deleteNoteAction` (`apps/web/src/lib/actions/notes.ts:56-64`).
- That means any other caller of `deleteNote` (e.g. `card-images.test.ts:50`
  calls it directly) silently leaves orphaned embedding rows. It also means
  the invariant "delete a note → its embeddings go with it" lives in the
  action layer, not the repo layer where the rest of the cascade logic lives
  — an easy place for a future caller to get it wrong.
- The "receipts cascade" test (`graph-receipts.test.ts`) never asserts
  anything about the `embeddings` table, so this gap is untested.

**I'm sending a subagent to fix this**: move the embeddings cleanup into
`deleteNote` itself (so it can't be forgotten by any caller), and add a test
asserting embeddings are gone after both `deleteNote` and `forgetContact`.
Small, self-contained, no live API keys needed. Will commit locally, not push.

## Update — cross-device capture audit (voice dictation / camera / card-photo upload), commit `5602e27`

Asked to confirm the web app's voice dictation, camera capture, and card-photo
upload genuinely work across laptop and phone browsers, not just whichever one
was tested first. No browser-automation tool available here either, so this
was a code-reading + current-facts research pass, not a click-through.

**Already correct, no change needed:**
- Voice dictation (`useDictation.ts`): all three callers (`AddNoteForm`,
  `QuickAddDock`, `SearchInput`) already gate the mic button behind the
  `supported` flag and hide it entirely when unsupported — no dead button.
  Confirmed via research (not stale training data) that this gating matters
  in practice: Firefox has never shipped `SpeechRecognition` for end users
  (flag-gated only), while Safari 14.1+ (macOS) / 14.5+ (iOS) supports it
  under the `webkitSpeechRecognition` prefix, which the existing `getCtor()`
  check already covers.
- `capture="environment"` on the card-photo file input: confirmed this is a
  mobile-only hint that desktop browsers simply ignore, falling back to a
  normal OS file picker with the `image/*` filter intact — genuinely
  graceful, and the "Take or choose a card photo" label is accurate either way.
- `downscalePhoto`'s EXIF-orientation handling turned out to be a non-issue,
  but only because of how the pieces fit together: every photo path (file
  upload, `QuickAddDock`'s upload button, live webcam capture) always passes
  through `PhotoCropper` before `downscalePhoto` ever sees bytes.
  `PhotoCropper` draws from an `<img>` element, which all evergreen browsers
  auto-rotate per EXIF before decode — so the cropped canvas output is always
  already correctly oriented with no EXIF tag left to misinterpret. Worth
  knowing this is a structural property of the pipeline order, not something
  `downscalePhoto` itself guards — if a future change ever calls
  `downscalePhoto` on a raw pre-crop file, the orientation bug would return.
- `WebcamCapture` already has `playsInline` + `muted` (required for iOS Safari
  inline autoplay) and uses `facingMode: "environment"` as a soft/ideal
  constraint, so it degrades gracefully to a laptop's only (front) camera
  instead of throwing `OverconstrainedError`.

**One real gap found and fixed:** `PhotoCropper.tsx` had no `onError` handler
on its `<img>`. Confirmed via current research that Chrome/Firefox/Edge have
*never* supported decoding HEIC/HEIF in `<img>` on any OS (patent-encumbered
HEVC codec) — only Safari can, by delegating to the OS's licensed decoder.
iPhones default to HEIC for camera photos. So the "scan the card with your
phone, upload the file from your laptop" flow this app explicitly supports —
if that laptop isn't running Safari — silently broke: the image never loads,
`rect` never gets set, "Use photo" stays disabled forever, with zero
explanation. Fixed with a minimal `onError` → clear message + "choose another
photo" affordance (no HEIC transcoding library added — that'd be
over-engineering for this fix). Typecheck/lint/vitest all green.
